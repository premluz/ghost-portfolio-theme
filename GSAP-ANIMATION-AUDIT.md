## What's Working Well ✅

### 1. Stage-Based Pinning Pattern (Operating Model, Hero Sequence)

Both follow the industry-standard **"tall container + pinned inner stage"** pattern:

```
Container height = N × 100vh (scroll budget)
Inner stage      = 100vh (what gets pinned)
ScrollTrigger    → pin: true, end: += N × 100vh
Separate scrub   → animation drives the master timeline
```

This is correct — two separate ScrollTrigger instances (one for pinning, one for scrubbing) is the intended GSAP pattern and prevents timing coupling issues.

### 2. CONSTANTS.js — Centralised Config

All major timing/easing values for pinned sections live in `CONSTANTS.js → SCROLL_ANIMATION_PAUSES`. Sections read from this at runtime via `window.SCROLL_ANIMATION_PAUSES`. This is good — single source of truth for:
- `HERO_PHRASE_PAUSES` per phrase
- `OPERATING_MODEL_PAUSE_PERCENTAGE`
- Ease functions for entrance/exit

### 3. Timeline Labels (Operating Model)

`operating-model.hbs` properly uses GSAP timeline labels (`item-0-start`, `entrance-end`, `pause-start`, etc.). Good for debugging, snap waypoints, and future modifications.

### 4. `prefers-reduced-motion` Handling

`initLogoMorphToOperatingModel()` correctly checks `window.matchMedia('(prefers-reduced-motion: reduce)')` and jumps to final state, skipping scroll animation.

### 5. Resize Debounce

Both `operating-model.hbs` and the resize handlers use debounced `setTimeout(() => ScrollTrigger.refresh(), 250)`. Correct pattern.

## Issues Found ⚠️

### 1. DUPLICATE SCROLLTRIGGER REGISTRATION

**`main.js` line 12** registers ScrollTrigger globally:
```js
gsap.registerPlugin(ScrollTrigger);
```

**`main.js` → `initStatsScroll()`** re-registers it conditionally:
```js
gsap.registerPlugin(ScrollTrigger);  // duplicate
```

GSAP handles duplicate registration gracefully but it signals code drift. The `initStatsScroll` guard block is overly defensive.

### 2. GSAP AVAILABILITY GUARD — INCONSISTENT PATTERNS

Three different approaches across the codebase:

```js
// Pattern A — retry loop (logo morph, operating model, hero sequence)
if (typeof gsap === 'undefined') { setTimeout(fn, 50); return; }

// Pattern B — silent bail (heading-animations, stacked cards)
if (typeof gsap === 'undefined' || typeof ScrollTrigger === 'undefined') return;

// Pattern C — elaborate conditional (initStatsScroll)
const hasScrollTrigger = typeof ScrollTrigger !== 'undefined' || gsap?.plugins?.ScrollTrigger;
```

Pattern A is correct for animation-critical sections (they will not work without GSAP). Pattern B is fine for progressive-enhancement components. Pattern C is overly complex and should be simplified.

### 3. CONSOLE.LOG LEAKAGE IN PRODUCTION

`hero-sequence.hbs` still has active `console.log` calls in scroll hot-paths:

```js
console.log('[HeroSeq] gsap.set opacity:0 on phrase', ...);
console.log('[HeroSeq] Phrase', i, 'entrance enter');
console.log('[HeroSeq] Init phrase', i, 'opacity:', ...);
```

And `initStatsScroll` in `main.js`:
```js
console.log('[stats-scroll] Initializing...');
console.log('[stats-scroll] Setting up ScrollTrigger...');
console.log('[stats-scroll] ✅ ScrollTrigger initialized');
```

These fire on every scroll and page load, slowing down scroll performance on mobile.

### 4. HERO SEQUENCE — BROKEN `speedFactor` VARIABLE

In `hero-sequence.hbs` the `speedFactor` variable is declared but **never used**:

```js
// Lines ~400, ~420 (both entrance and exit builders)
const speedFactor = (window.SCROLL_ANIMATION_PAUSES?.HERO_PHRASE_SPEEDS?.[i] || 1);
// speedFactor is never applied to any duration or scroll distance
```

`CONSTANTS.js` defines `HERO_PHRASE_SPEEDS: [1, 1, 0.8, 0.8]` but the hero sequence completely ignores it. Either apply it or remove both.

### 5. STATEMENT SECTION — ORPHANED CODE

`statement-slide-logos.hbs` is now only 22 lines (an `#helix` section with headings) but:
- `GSAP-CONFIG.md` still documents it as a 3-phase pinned sequence with logos ribbon
- `CODEBASE_ANALYSIS.md` still references `statement-slide-logos.hbs` as having complex JS
- `GSAP-CONFIG.md` section "Statement Section Deep Dive" describes an entirely different component

The `logos-ribbon.hbs` partial still exists and presumably has its own animation. The old complex statement section has been removed. Docs are stale.

### 6. `heading-animations.js` — DUAL SYSTEM WITH SHARED SPLITMAP

The refactored `heading-animations.js` now splits candidates into `scrubHeadings` (h2/h3) and `observerHeadings` (h1/h4), but both paths write to the **same** `splitMap` and `wordTlMap`. If a heading somehow matches both paths this would be a bug. The maps should be type-scoped or the comment should explicitly note this cannot happen.

### 7. `initLogoMorphToOperatingModel` — RETRY LOOP VS LOAD EVENT

The logo morph uses a `setTimeout` retry pattern:
```js
if (typeof gsap === 'undefined') { setTimeout(fn, 50); return; }
```

All other section-specific scripts use `window.addEventListener('load', fn, { once: true })` which is more reliable than polling. This is inconsistent and creates unpredictable timing relative to other scroll triggers.

### 8. TWO SCROLL LISTENERS FOR NAV HIDE

`ui-utilities.js` uses a `requestAnimationFrame` batched scroll listener. But `heading-animations.js` and `scroll-scrub-anim.js` also add their own scroll-related listeners. While not strictly duplicate, these compound in scroll-heavy scenarios.

## Optimisation Opportunities 🔧

### A. Unify GSAP Guard into a Single Utility

```js
// utils/wait-for-gsap.js
function whenGSAPReady(fn) {
  if (typeof gsap !== 'undefined' && typeof ScrollTrigger !== 'undefined') {
    fn();
  } else {
    window.addEventListener('load', fn, { once: true });
  }
}
```

Replace all `setTimeout(fn, 50)` retry loops with this. Eliminates race conditions.

### B. Fix `speedFactor` in Hero Sequence

Either apply it:
```js
// In entrance builder scroll trigger:
end: `top+=${sliceStart + SCROLL_PER_PHRASE * 0.4 / speedFactor} top`,
```

Or remove `HERO_PHRASE_SPEEDS` from `CONSTANTS.js` and the hero sequence.

### C. Remove Active Console Logs from Hero Sequence

```js
// hero-sequence.hbs — remove or comment these:
console.log('[HeroSeq] gsap.set opacity:0 on phrase', ...);
console.log('[HeroSeq] Phrase', i, 'entrance enter');
console.log('[HeroSeq] Init phrase', i, 'opacity:', ...);
console.log('[HeroSeq] buildEntrance fade', ...);
console.log('[HeroSeq] buildExit fade', ...);
```

### D. Logo Morph — Use Load Event Instead of Retry Loop

```js
// Replace:
function initLogoMorphToOperatingModel() {
  if (typeof gsap === 'undefined') { setTimeout(initLogoMorphToOperatingModel, 50); return; }
  // ...
}

// With:
window.addEventListener('load', function initLogoMorphToOperatingModel() {
  // gsap is guaranteed available at load
}, { once: true });
```

### E. Stats Scroll — Simplify Guard

```js
// Replace the 10-line conditional block:
const hasScrollTrigger = typeof ScrollTrigger !== 'undefined' || gsap?.plugins?.ScrollTrigger;
// ... duplicate register logic

// With:
if (typeof gsap === 'undefined' || typeof ScrollTrigger === 'undefined') return;
```

## Pinned Section Pattern — Reference

The correct, consistent pattern (used well in `operating-model.hbs`):

```js
// 1. Set initial state
gsap.set(elements, { opacity: 0, x: '-100%' });

// 2. Pin the stage (separate ST instance)
ScrollTrigger.create({
  trigger: stage,
  start: 'top top',
  end: `+=${TOTAL_SCROLL}`,
  pin: true,
  pinSpacing: true,
  anticipatePin: 1,
});

// 3. Master timeline (paused)
const tl = gsap.timeline({ paused: true });
// ... add tweens with labels

// 4. Scrub timeline drives master (second ST instance)
ScrollTrigger.create({
  trigger: stage,
  start: 'top top',
  end: `+=${TOTAL_SCROLL}`,
  scrub: 1,
  animation: tl,
});

// 5. Resize handler
window.addEventListener('resize', debounce(() => ScrollTrigger.refresh(), 250));
```

**Logo morph uses `toggleActions`** — this is correct for a non-scrubbed animation that simply plays/reverses on enter/leave. It does NOT need a master timeline pattern. It's the right tool for that job.
